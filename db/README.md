# Database layer (PostgreSQL 16 + PostGIS)

This is the heart of the dashboard's data governance. Everything the public sees
passes through **one gate**: the `public_occurrences` view.

## Layout

```
db/
├── migrations/          Applied in order by `make db-migrate` (idempotent)
│   ├── 0001_extensions.sql            PostGIS
│   ├── 0002_core_occurrences.sql      occurrences (precise/private) + public_config
│   ├── 0003_sensitive_species.sql     the control list that drives the gate
│   ├── 0004_public_occurrences_view.sql   THE GATE (functions + view)
│   ├── 0005_tile_functions.sql        Martin tile function (reads the gate only)
│   ├── 0006_roles.sql                 read-only role; no access to precise data
│   └── 0007_indexes.sql               indexes on the precise base table
├── optional/
│   └── 0100_public_occurrences_mv.sql Cached materialized copy for large data
├── seed/                Synthetic demo data ONLY (safe to commit)
│   ├── sensitive_species_demo.sql
│   └── seed_synthetic.sql             generated by scripts/seed_synthetic.py
├── tests/
│   └── test_generalisation_gate.sql   the fail-closed assurance test
└── functions/           (docs pointer; functions live in the migrations)
```

## The one thing to understand

```
occurrences (precise, private)                 <- app role has NO access
     │
     │  public_occurrences VIEW:
     │    * generalise_point()  -> snap to a coarse grid cell (EPSG:27700), fail-closed
     │    * drop recorder1 + all PII and precise columns
     │    * omit blocked taxa; default-generalise anything unconfirmed
     ▼
public_occurrences (generalised, safe)         <- app role reads ONLY this
     ├── Martin tile function  public_occurrences_mvt()
     └── FastAPI read queries
```

A PostgreSQL view accesses its underlying tables with the **view owner's**
privileges. The application connects as `brerc_readonly`, which is granted
`SELECT` on `public_occurrences` but has that grant **revoked** on
`occurrences`. So the app can read generalised data through the view yet
physically cannot read precise coordinates or recorder names — even if the API
code had a bug. That is the enforcement.

## Resolution rules (what precision a record is shown at)

The effective public resolution for a record is the **coarsest** (largest) of:

1. `public_config.baseline_resolution_m` — a floor applied to *every* public
   record (default 1 km / monad). Nothing is ever shown finer publicly.
2. the record's own grid-reference precision — we never claim more precision than
   the source actually has.
3. the sensitive tier, if the taxon is listed: its `required_resolution_m`, or
   the `review_default_resolution_m` if it is listed but unspecified.

If the taxon is `public_blocked`, or the geometry is missing, the record is
**dropped from the public tier entirely** (fail-closed). See
[`../docs/SENSITIVE_SPECIES.md`](../docs/SENSITIVE_SPECIES.md).

## Running it

```bash
make up            # starts Postgres/PostGIS (and the rest of the stack)
make db-migrate    # apply 0001..0007
make db-seed       # load the synthetic sensitive list + synthetic records
make gate-test     # prove the gate holds
```

To change how coarse the public data is, edit the single `public_config` row —
no code change (`UPDATE public_config SET baseline_resolution_m = 2000;`).

## Assumptions to confirm with BRERC

The `occurrences` columns marked `[assumed]` in `0002` are a reasonable reference
schema; swap them for BRERC's actual column list when known. `Recorder1` and
`BLISS` are confirmed BRERC fields (stored lower-cased with a documented COMMENT
mapping). The sensitive list in `seed/` is synthetic — replace it with BRERC's
authoritative NBN / statutory list and resolutions.
